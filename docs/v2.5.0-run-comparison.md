# v2.5.0 — Run Comparison (A vs B)

> Audience: STNH Log Analyzer dev team and modders running balance experiments.
> File touched: `index.html` only (single-file app — no build step, no dependencies).

## Why this exists

Modders testing balance changes (e.g. Orion's `STH_fix.300` event that gives extra `district_generator` districts to specific AI-empire categories) need to tell whether their tweak actually moved the needle, or whether the difference between two playthroughs is just normal run-to-run variance.

Until v2.4, the analyzer **merged** every uploaded log into a single global `data` object. Two logs in → numbers for the same date were summed or overwritten. There was no way to hold "Run A" and "Run B" side by side.

v2.5 introduces multi-run support and a Spreadsheet view that shows per-empire deltas between any two runs at any common date, with per-category averages so the headline question — *"Did Significant empires actually gain dilithium in Run B vs Run A?"* — has a one-row answer.

## What changed for users

### Upload flow
A new checkbox appears on the upload screen **only when 2+ files are queued** for analysis (pending uploads + library selections combined):

> **☐ Merge all files into a single run (no A/B compare)**

- **Unchecked (default)**: each file becomes its own run. The first run drives all the existing tabs (Revenue, Stats, Chart, Wars, Faction Detail, etc. — unchanged); the additional runs are available in the new Spreadsheet compare mode.
- **Checked**: legacy behavior. Everything is merged into one run, exactly as before. The compare mode stays hidden.

Single-file uploads are unaffected — no checkbox shown, behavior identical to v2.4.

### Spreadsheet → Compare runs (A vs B)
The Spreadsheet tab's **Timeframe** dropdown gets a new option that's only visible when ≥2 runs are loaded:

> **Compare runs (A vs B)**

Selecting it reveals three additional controls:
- **Run A** — picks the baseline run.
- **Run B** — picks the comparison run.
- **Date** — only dates present in *both* runs are listed.

The result table has five columns:

| Empire | Run A | Run B | Δ (B−A) | Δ% |
|--------|------:|------:|--------:|----:|

Below the per-empire rows, one summary row per faction category shows the **average** value across all empires of that category in each run, plus the average delta. This is the row that answers the modder's actual question.

All other Spreadsheet controls keep working in compare mode:
- **Resource/Stat select** — switches which metric is being compared.
- **Search** — filters by empire name.
- **Hide zero-only** — drops empires that are zero in both runs.
- **Heatmap** — Δ column gets a red ↔ green gradient (red = empire did worse in Run B, green = better). Intensity is normalized to the largest absolute Δ on screen.
- **Sort by**: Name / Latest Value (= Run B) / Total (= A+B sum) / Growth (= Δ).
- **Category filter bar** (the one added in v2.4) — applies to compare mode unchanged via the existing `sheetFactionPasses()` helper.
- **Export CSV** — produces a 6-column file: `Empire, Category, Run A: <name>, Run B: <name>, Delta, Delta %`.

Empires that exist in only one run get `—` for the missing side; Δ and Δ% are blank for that row.

## Architecture

### `runs[]` and the global `data` mirror
Single source of truth lives in:

```js
let runs = []; // [{ id: 'r0', name: 'run1.log', data: <bucket> }, ...]
```

Each `bucket` has the exact same shape as the existing global `data` object (see `freshDataObj()`). To avoid a sweeping refactor of the 11 existing log-mode tabs (which all read `data.revenue`, `data.stats`, etc.), the global `data` is **mirrored from `runs[0]`** after every analyze. This is done by `loadIntoGlobalData(src)`, which reassigns each property of the (`const`) `data` object in place.

Multi-run consumers (currently only `renderSheetCompare()`) read directly from `runs[i].data` and never touch the global `data`.

### `parseLog()` / `finalizeData()` signatures

```js
function parseLog(text, fileName, target = data) {
  const data = target; // shadows outer module-level `data`
  /* ...body unchanged... */
}
function finalizeData(target = data) {
  const data = target;
  /* ...body unchanged... */
}
```

The `target = data` default parameter is evaluated in the parameter scope (which sees the outer module-level `data`), then the body shadows it with `const data = target`. Net effect: existing call sites that pass two args still write to the global `data`; new call sites pass a `freshDataObj()` bucket as the third arg and parse into that.

### Analyze handler branch
In the `analyzeBtn` click handler:

```js
const mergeAll = document.getElementById('mergeRunsCheck').checked || totalCount < 2;

if (mergeAll) {
  // One bucket, all files parsed into it, then mirrored to global data.
  const bucket = freshDataObj();
  for (const p of parseList) parseLog(p.content, p.name, bucket);
  finalizeData(bucket);
  loadIntoGlobalData(bucket);
  runs.push({ id: 'r0', name: '...', data: bucket });
} else {
  // One bucket per file. runs[0] is mirrored to global data; runs[1..] are compare-only.
  for (let i = 0; i < parseList.length; i++) {
    const bucket = freshDataObj();
    parseLog(p.content, p.name, bucket);
    finalizeData(bucket);
    runs.push({ id: 'r' + i, name: p.name, data: bucket });
  }
  loadIntoGlobalData(runs[0].data);
}
```

The auto-analyze fallback in `stageFiles()` (the path triggered when a `.log` is dropped while already in save mode) was updated to follow the same pattern and always create at least `runs[0]`.

### `renderSheetCompare()`
- Reads run A and run B straight from `runs[ai].data` and `runs[bi].data`.
- Builds the empire union from both runs at the chosen date.
- Resolves each empire's category by checking `runA.data.factionCategories[date]` first, then `runB.data.factionCategories[date]`, then falling back to `getCategoryForFaction(f)` (which uses the global mirror).
- Reuses `sheetFactionPasses()` for the category-filter bar, and `sheetExportCache` for CSV export.
- Heatmap uses RGB values for `--success` (78,205,196) and `--accent` (233,69,96) inlined, since CSS variables can't be interpolated into `rgba()`.

### State reset
`resetToUpload()` now clears `runs = []`, hides the merge checkbox, and unchecks it. Everything else there was already correct.

## Testing

No automated test suite in the repo, so verify manually. From the repo root:

```sh
python serve.py     # serves on http://localhost:8080
```

### Regression
1. **Single file**: load 1 `.log` → no merge checkbox appears, Spreadsheet has no "Compare runs" option, all 11 tabs render as before.
2. **Merged 2-file**: load 2 `.log` files, **check** "Merge all files…", Analyze → behaves identically to v2.4 (no compare option, data summed into one bucket).

### Happy path
3. **A/B compare**: load 2 `.log` files, **leave checkbox unchecked**, Analyze.
   - Spreadsheet → Timeframe dropdown shows "Compare runs (A vs B)".
   - Selecting it reveals Run A / Run B / Date selects, prefilled with the two logs and a common date.
   - Δ column has red/green heatmap; Δ% is signed and percent-formatted.
   - Per-category summary rows at the bottom show `Ø Major (n)`, `Ø Significant (n)`, etc.
   - Switching the Resource dropdown updates the table and re-filters the Date dropdown to common dates only.
   - Category-filter bar (e.g. include only `Significant`+`Medium`) filters both per-empire rows and the per-category summary.
   - "Export CSV" produces a 6-column file with `Run A: <name>` / `Run B: <name>` headers.

### Edge cases
4. **Empire only in one run**: shows `—` in the missing column; Δ and Δ% blank.
5. **Same run picked for A and B**: info bar says "Run A and Run B are the same — pick different runs".
6. **No common date**: info bar says "No common date between the two selected runs".
7. **Reset**: click "New Analysis" → upload screen, merge checkbox hidden, `runs.length === 0`.
8. **3+ runs**: load 3 `.log` files unchecked → all three appear in Run A / Run B dropdowns; user can compare any pair.

### Browsers
Test in a Chromium-based browser via both `file://` and `http://localhost:8080` (the app must work from both because it's served on GitHub Pages and also opened directly).

## Non-goals (deliberately out of scope)

- **3-way compare** (Run A vs B vs C). The current UI only compares pairs. If needed later: turn the two run selects into a multi-select and add columns dynamically.
- **Time-series compare** (Δ over a date range, not just a single date). Could be a follow-up; the data is already in `runs[i].data`.
- **Cross-run save support**. This release only adds run support to log mode. Save mode is still single-save.
- **Auto-detection of "significant" deltas**. The Insights tab still operates only on the first run.

## Known caveats

- Each run keeps a full copy of its parsed data in memory. With many large logs (`>50 MB` each), watch memory usage. The IndexedDB log library still stores raw text, not parsed buckets, so this only matters during a single session.
- The merge checkbox is intentionally **off** by default. Users who relied on the old "drop everything in" behavior with multi-file uploads need to learn one new click. Consider this when answering support questions.
- `loadIntoGlobalData()` does **shallow** assignment of the bucket's properties to `data`. If new top-level keys are ever added to the data shape, both `freshDataObj()` and `loadIntoGlobalData()` need to learn about them — keep them in sync.
