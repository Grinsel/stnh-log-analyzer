# v2.6.0 – v2.7.1 — Orion's new logging events

> Audience: STNH Log Analyzer dev team, modders, and the STNH logging-events contributor.
> Files touched: `index.html`, `CHANGELOG.md`, `README.md` (new), `docs/` (this file).

## Why this exists

On 2026-04-24, Orion1988 shipped a batch of new structured logging events in `STH_test_events.txt` and asked:

1. How much data can the analyzer handle overall?
2. Should all of the new events be wired up, or only the high-signal ones?

The new events:

| Event | Payload suffix | Notes |
|---|---|---|
| Fleet power | `<faction> <value> fleet power` | Country military strength (like the observer outliner) |
| Unemployed pops | `<faction> <value> unemployed pops estimate` | Estimate, diverges from in-game UI by tens/hundreds |
| Influence revenue | `<faction> <value> influence revenue` | The one missing major resource |
| Major income | `<faction> <value> <resource> income` | Energy/Mineral/Food/Alloy/Unity/Engineering/Physics/Society/Dilithium |
| Rare income | `<faction> <value> <resource> income [AI only]` | 17 rare keys (latinum, water, brizeen, …) |

Key finding from testing Orion's `game.log` (5 MB, ~27k lines): **`revenue` and `income` are not duplicates**. Revenue is gross production, income is net after upkeep. Example — Klingon Empire on 2180.01.01: **+365 energy revenue vs. -67 energy income**. That 432-unit delta is upkeep burden, previously invisible in any tab.

**Decision:** wire up all 5 event types, plus derive `unemployment %` (= unemployed / owned pops × 100) per Orion's suggestion.

## What changed for users

### v2.6.0 — parser + stats coverage

All five event types are now parsed. Four of them slotted into existing UI via the `RESOURCES` / `STATS` constants and immediately appeared across Stats, Rankings, Chart, Spreadsheet, Faction Detail, Compare, and Timeline tabs:

- **Fleet Power** stat — first real military-strength metric in the app.
- **Unemployed Pops Estimate** stat + derived **Unemployment %**.
- **Influence revenue** joined the resource family.

Income data (both major and rare) was stored in a new `data.income` stream but only surfaced as **counts** on the Dashboard and per-file counts in the Meta tab. The numbers themselves stayed invisible until v2.7.0.

### v2.7.0 — Income tab + scalable date picker

A dedicated **Income tab** sits between Revenue and Stats. Controls and interaction mirror the Revenue tab (search, category quick-select, checkbox-to-graph, CSV export) so the learning curve is zero.

**Sub-view toggle** above the table picks the column set:
- **Major (9)** — the nine primary economy resources. Default.
- **Rare (17)** — all strategic resources.
- **All (26)** — everything at once, with horizontal scroll.

The chart overlay is income-aware: selecting empires and hitting "Show Selected Empires Graph" opens a line chart with `chartState.mode = 'income'`. The title reads `<Metric> Income (net) — N Empires`. The metric dropdown is filled from the current sub-view's keys.

**Empty state:** loading a pre-April-2026 log (no income events) shows a helpful message explaining the mod-version requirement instead of a blank table.

### Scalable date picker

Orion's current logging is decadal (2160 / 2170 / 2180 — three snapshots over 30 years). If Russ changes the interval to annual or monthly, a 200-year playthrough would flood a plain `<select>` with 200 or 2400 entries. The Income tab uses a new reusable control, `createDatePicker()`, that stays usable at any scale:

```
Date: [2180.01.01 ▼]  ◀ ▶  [────────●────────]   [2150.01.01 — 2180.01.01 · 3 snapshots]
      combobox         prev/next  slider (≥10)   span label
```

- **Combobox** (`<input list>` + `<datalist>`) — type-ahead filter. Browser-native, handles thousands of options without lag.
- **Prev/Next** buttons — step to the previous/next available snapshot (disabled at boundaries).
- **Range slider** — drag to scrub through time. Auto-hidden when fewer than 10 snapshots exist.
- **Span label** — shows the full range and snapshot count so the user sees the logging density at a glance.

All four sub-controls stay in sync. Invalid combobox text reverts to the last valid value.

## Capacity — answer to Orion's first question

No hard limit on the analyzer side. Everything is browser-side single-file JavaScript:

- 5 MB game.log: trivial.
- 50–100 MB logs: comfortable.
- Gigabyte-scale logs would need a streaming parser refactor, but we're not remotely near that.

Multiple runs persist via IndexedDB (`stnh_log_analyzer` DB, `logs` store) for cross-session reuse.

## Architecture

### Data model

```js
const data = {
  revenue: {},    // { date: { faction: { resource: value } } } — GROSS, unchanged
  income:  {},   // { date: { faction: { key: value } } }       — NET, new in v2.6.0
  stats:   {},   // { date: { faction: { metric: value } } }    — extended with fleet power, unemployed, unemployment %
  /* ...rest unchanged... */
  counts:  { revenue, income, stats, warDecl, invasion, warStatus, human, difficulty, factionCategory }
};
```

`INCOME_KEYS` lists all 26 parsable income resource keys. `INCOME_SHORT` maps each to a display label. Longest-first suffix matching is used so multi-word keys like `ketracel white` don't get mis-matched as `white`.

### Parser

`parseLog()` got one new branch, positioned after the stats-matching block:

```js
{
  let payloadForIncome = payload;
  if (payloadForIncome.endsWith(' AI only')) {
    payloadForIncome = payloadForIncome.slice(0, -' AI only'.length);
  }
  if (payloadForIncome.endsWith(' income')) {
    const core = payloadForIncome.slice(0, -' income'.length);
    for (const key of incomeSorted) {
      const suffix = ' ' + key;
      if (core.endsWith(suffix)) {
        /* extract faction + value, dedupe per (date, faction, key) */
      }
    }
  }
}
```

**Dedup:** STH_test_events currently emits each income key twice per date (lines 1772 and 1824 for dilithium income, for example). The parser keeps the first value and silently drops subsequent duplicates for the same `(date, faction, key)` tuple. This is reported to Russ as a log-size optimization.

### Derived metric: `unemployment %`

`finalizeData()` walks `data.stats` after parsing and computes:

```js
const unemp = m['unemployed pops estimate'];
const pops  = m['owned pops'];
if (typeof unemp === 'number' && typeof pops === 'number' && pops > 0) {
  m['unemployment %'] = (unemp / pops) * 100;
}
```

It's stored inside the same stats bucket as a regular stat so all existing tabs (Rankings, Chart, Spreadsheet, Faction Detail) pick it up with no per-tab changes.

### Income tab render pipeline

Functions added (all sit between `renderRevenue` and the CHART section):

- `initIncome()` — wires the sub-toggle buttons, search input, graph button; creates the date picker; handles the empty-state.
- `renderIncome()` — 1:1 port of `renderRevenue()` with `data.income[date]` as source and `incomeKeysForView()` driving columns.
- `exportIncomeCSV()` — filename includes the active sub-view: `income_major_2180.1.1.csv`.
- `updateIncomeSelectedUI()` — updates the "N selected" counter and graph-button disabled state.
- `openIncomeChart()` — sets `chartState.mode = 'income'`, fills the metric dropdown from the current sub-view, then opens the existing chart overlay.

State:

```js
let incomeSortCol = 0, incomeSortAsc = true;
const incomeSelected = new Set();
let incomeSubView = 'major';      // 'major' | 'rare' | 'all'
let incomeDatePickerCtrl = null;  // handle returned by createDatePicker()
```

All four reset in `resetToUpload()`.

### `createDatePicker(container, { dates, onChange, initial })`

Returned handle:

```js
{ getValue(), setValue(d), setDates(newDates, newInitial) }
```

Implementation lives near the DATE UTILITIES (index.html ~2670). It injects HTML into `container`:

```html
<input type="text" class="dp-combo" list="<generated-id>" spellcheck="false">
<datalist id="<generated-id>"></datalist>
<button class="dp-nav" data-dir="-1">◀</button>
<button class="dp-nav" data-dir="1">▶</button>
<input type="range" class="dp-slider" min="0" max="N-1">
<span class="dp-span"></span>
```

Invariants maintained internally:
- `current` is always either a date in `dates` or `null` (when `dates` is empty).
- Combobox accepts both raw `2180.1.1` and formatted `2180.01.01`; anything else reverts to last valid.
- Slider hidden when `dates.length < 10`.
- Placeholder `"type to filter..."` shown when `dates.length >= 100`.
- Prev/Next buttons disabled at range boundaries (no wrap-around).

### Chart `source` switch

`drawChart()` was a two-way branch (revenue vs. stats). Now three-way:

```js
const source =
  mode === 'stats'  ? data.stats :
  mode === 'income' ? data.income :
                      data.revenue;
```

Labels adjusted so the title reads `<Metric>`, `<Metric> Revenue`, or `<Metric> Income (net)` depending on mode.

## Upstream follow-up for Russ

The `… income` events are **double-logged** per date: once at STH_test_events line 1772 and again at line 1824 for dilithium, with analogous pairs for the other keys. The analyzer dedupes (first value wins), so there's no data corruption — but the logs would be roughly half the size if one of the two emissions is dropped in the source.
